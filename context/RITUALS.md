# RITUALS.md — 8 обязательных ритуалов проекта GSM

## Ритуал 1: Начало сессии

**Trigger:** Каждый раз при старте новой сессии с AI.

**Шаги:**
1. `bash scripts/start-session.sh` — показать git log, статус, docker ps
2. Прочитать `context/RULES.md` — освежить стек и правила
3. Прочитать `context/SNAPSHOT.md` — текущее состояние
4. Прочитать `context/BACKLOG.md` — определить следующую задачу
5. `docker compose ps` — все ли контейнеры в Up
6. Доложить пользователю: "Проект в состоянии X. Предлагаю сделать Y."

**Цель:** AI входит в контекст за 15 секунд, не теряя нитки.

---

## Ритуал 2: Конец сессии

**Trigger:** Перед завершением работы.

**Шаги:**
1. `find backend -name "*.py" -exec python3 -m py_compile {} \;` — проверить синтаксис
2. `bash scripts/update-docs.sh` — автообновление документов
3. `git status` — проверить что всё учтено
4. `git add -A && git commit -m "docs: ..." && git push origin main`
5. Доложить: "Закоммичено: ..., отправлено в main"

**Цель:** Ни одна сессия не заканчивается без коммита и обновлённой документации.

---

## Ритуал 3: Архитектурный Gate

**Trigger:** Перед созданием нового модуля, сервиса, роутера или таблицы.

**Шаги:**
1. Прочитать `context/ARCHITECTURE.md`
2. Ответить на вопросы:
   - Соответствует ли новый модуль Modular Monolith?
   - Добавлен ли `company_id` для multi-tenancy?
   - Нужна ли новая таблица в БД?
3. Если отклонение от архитектуры — запросить подтверждение у пользователя

**Цель:** Предотвратить разрастание хаоса и овер-инжиниринг.

---

## Ритуал 4: LLM-самопроверка

**Trigger:** После каждой генерации кода (перед сохранением).

**Шаги:**
1. Проверить что код соответствует `context/RULES.md`:
   - TypeHints везде?
   - Pydantic V2 стиль?
   - async/await для БД?
   - Обработка ошибок?
2. Проверить что нет `# TODO`, `pass`, плейсхолдеров
3. Проверить что RLS/tenant_id не забыты

**Цель:** Не дать AI сгенерировать «грязный» код.

---

## Ритуал 5: Quality Gates (перед коммитом)

**Trigger:** Автоматически перед каждым коммитом (через pre-commit) и вручную перед `git push`.

**Шаги (строго по порядку):**
1. `ruff check --fix backend/app/ && ruff format backend/app/` — форматирование + линтинг
2. `mypy backend/app/ --ignore-missing-imports` — статическая типизация (если ошибки — коммит отклонён)
3. `python3 -m py_compile` всех изменённых `.py` — синтаксис
4. `pytest backend/tests/ -x -q` — юнит-тесты (если есть)
5. Если любой шаг FAIL — код НЕ коммитится

**Установка инструментов:**
```bash
source .venv/bin/activate
pip install ruff mypy
```

**Цель:** В репозиторий не попадает код с ошибками типов, стиля или синтаксиса.

---

## Ритуал 6: Миграционный Gate

**Trigger:** Любое изменение `docker/init.sql` или добавление Alembic-миграции.

**Шаги:**
1. Убедиться что изменения обратно совместимы (только ADD, без DROP/RENAME без необходимости)
2. Проверить что новые таблицы имеют `company_id` (если tenant-aware)
3. Проверить что добавлены индексы на новые колонки
4. Проверить что `init.sql` остаётся идемпотентным (`CREATE IF NOT EXISTS`)

**Цель:** Никогда не потерять данные клиентов из-за миграции.

---

## Ритуал 7: Security Gate

**Trigger:** Изменение любого кода, связанного с auth, billing, RLS, или добавление нового эндпоинта.

**Шаги:**
1. Нет ли хардкода паролей/токенов?
2. RLS включена на новой таблице?
3. JWT проверяется (а не только Bearer-заголовок)?
4. Пароли хэшируются bcrypt?
5. Grace Period не нарушен?

**Цель:** Zero-trust: даже внутренний код не должен иметь доступа к чужим данным.

---

## Ритуал 8: Релизный Gate

**Trigger:** Перед созданием Git-тега (`v*`).

**Шаги:**
1. Прочитать `context/VERSIONING.md` — определить MAJOR/MINOR/PATCH
2. Проверить что все Quality Gates пройдены
3. `docker compose build --no-cache` — чистая сборка
4. Smoke test: запустить контейнеры, проверить healthcheck
5. `git tag vX.Y.Z && git push origin vX.Y.Z`

**Цель:** Каждый тег — гарантированно рабочее состояние.
