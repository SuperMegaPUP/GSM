# VERSIONING.md — Конституция версионности

## 1. SemVer (Semantic Versioning)

Формат: `vMAJOR.MINOR.PATCH`

| Компонент | Когда увеличивать | Пример |
|---|---|---|
| MAJOR | Любое ломающее изменение API, БД, архитектуры | v1.0.0 → v2.0.0 |
| MINOR | Добавление функционала (новый модуль, эндпоинт) | v1.0.0 → v1.1.0 |
| PATCH | Исправление багов, оптимизация, документация | v1.0.0 → v1.0.1 |

## 2. Чек-лист перед тегом

- [ ] Все Quality Gates пройдены (ruff → pytest → build)
- [ ] `BACKLOG.md` обновлён (выполненные задачи помечены)
- [ ] `WORKLOG.md` содержит запись о сессии
- [ ] `SNAPSHOT.md` отражает актуальное состояние
- [ ] `CHANGELOG.md` (или `HISTORY.md`) обновлён
- [ ] Все изменения закоммичены и запушены
- [ ] `docker compose build --no-cache` проходит

## 3. Гарантии восстановимости

- Каждый тег должен соответствовать состоянию, когда `docker compose up -d` работает без ошибок
- `init.sql` должен быть идемпотентным или иметь обратную миграцию
- Для каждой версии в issues записаны известные проблемы (если есть)

## 4. Структура тегов

```
v0.1.0 — первый рабочий прототип (MVP)
v1.0.0 — стабильный релиз
v1.1.0 — новый модуль (Квиз, CRM-интеграция)
```
