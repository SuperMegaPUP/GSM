> **Цель:** Поддержка разных типов техники (легковые, грузовые, спецтехника, мото, бензопилы) в едином UI и БД без переписывания при добавлении новых типов.

## 🏗️ Архитектурное решение: Полиморфная модель

### Принципы

1. **Single Table Inheritance + JSONB** — единая таблица `vehicles` для всех типов, type-specific атрибуты в `attributes JSONB`
2. **Полиморфные узлы** — таблица `vehicle_nodes` хранит разные наборы узлов для каждого типа
3. **Рекомендации с рангом** — `recommendation_rank` (1=primary, 2=alt1, 3=alt2) для поддержки PVL-формата
4. **Strategy pattern в ETL** — каждый формат каталога имеет свой парсер, фабрика выбирает подходящий автоматически

### Схема данных

```
┌────────────────────────────────────────┐
│  vehicles (ВСЕ типы ТС)                │
│  ──────────────────────────────────    │
│  id UUID (PK)                          │
│  vehicle_type ENUM ← дискриминатор     │
│  brand, model, generation, sub_model   │
│  year_start, year_end                  │
│  market (JDM/EU/US/RU)                 │
│  attributes JSONB ← type-specific поля │
│  external_codes JSONB                  │
│  source (japan_catalog/pvl/manual)     │
│  source_hash UNIQUE                    │
└──────────────┬─────────────────────────┘
               │ 1:N
               ▼
┌────────────────────────────────────────┐
│  vehicle_recommendations               │
│  ──────────────────────────────────    │
│  id UUID (PK)                          │
│  vehicle_id → vehicles.id (FK)         │
│  node_code (полиморфный)               │
│  fluid_id → fluids.id (FK)             │
│  recommendation_rank (1/2/3) ← НОВОЕ   │
│  is_oem_recommendation                 │
│  volume_liters VARCHAR (диапазоны)     │
│  applicability_conditions JSONB ← НОВОЕ│
│  (для маркеров Diesel, AC, и т.д.)     │
└────────────────────────────────────────┘
               ▲
               │ N:1
┌──────────────┴─────────────────────────┐
│  vehicle_nodes (реестр узлов)          │
│  ──────────────────────────────────    │
│  vehicle_type + node_code UNIQUE       │
│  node_label, category_group, icon_name │
│  display_order                         │
└────────────────────────────────────────┘
```

## 🎨 UI-концепция: Tabs для переключения типа техники

**Не отдельное меню** для каждого типа, а **единая страница с tabs**:

```
┌──────────────────────────────────────────────────────────────────┐
│ [🚗 Легковые] [🚚 Грузовые] [🚜 Спецтехника] [🏍 Мото] [🪚 Бензопилы]│
└──────────────────────────────────────────────────────────────────┘
                              ↓
            ┌─────────────────────────────────┐
            │ Форма подбора (адаптивная)      │
            │ — Поля меняются по типу ТС       │
            │ — Узлы меняются по типу ТС       │
            │ — Результаты в FluidCard         │
            └─────────────────────────────────┘
```

### Преимущества подхода:
1. **Единый URL** (`/dashboard/search`) — не нужно плодить роуты
2. **Быстрое переключение** — пользователь может сравнить масла для легкового и спецтехники за секунду
3. **Органичное расширение** — добавили новый `vehicle_type` → добавили tab → готово
4. **Сохранение контекста** — при переключении типа сохраняются другие параметры (год, бренд)

### Динамические поля формы по типу ТС

| Тип ТС | Уникальные поля |
|---|---|
| Легковые | Код двигателя, Код кузова, Объём двигателя (л), Тип топлива |
| Грузовые | Полная масса (кг), Конфигурация осей (4x2, 6x4), Тип применения |
| Спецтехника | Тип оборудования, Эксплуатационная масса, Мощность (кВт) |
| Мото | Объём двигателя (см³), Тип мото (спорт/круизер/эндуро) |
| Бензопилы | Тип оборудования, Объём двигателя (см³), Длина шины |

## 🎯 Ключевые архитектурные решения

### ✅ Что сделано правильно

1. **Полиморфизм через JSONB** — позволяет добавлять новые типы техники без миграции схемы
2. **Strategy pattern в ETL** — каждый новый формат = новый класс-парсер, без изменения существующего кода
3. **`recommendation_rank`** — поддерживает PVL-формат с основным + 2 альтернативами
4. **`applicability_conditions`** — поддерживает маркеры комплектации (Diesel, AC, и т.д.)
5. **Backward compatibility** — старая таблица `car_variants` остаётся, есть view `car_variants_unified`

### ⚠️ Что нужно обсудить

1. **Миграция существующих данных** — нужно ли переносить 11 167 строк из `car_variants` в `vehicles`? Или оставить как есть и новые данные писать в `vehicles`?
2. **API endpoint** — сохранять ли `/api/v1/catalog/*` или сделать новый `/api/v2/vehicles/*`?
3. **Импорт 21 762 рекомендаций из PVL** — это **большой** объём, нужно либо Celery-задача, либо батчинг
